02 / Decision

Todo is a pane

One navigation model is enough for the rewrite: choose a workspace and arrange its panes. Todo stays visible beside the agent or shell executing its work.

UI rule

Todo is a focusable peer in WorkspaceLayout. Runtime rows show activity at rest, then expand into mute and close; Todo reveals close only.

Benefit

No new tabs or hidden task context while working in terminals.

Architecture cost

Todo needs a non-terminal pane kind. Start from existing daemon session todo events; do not create native-local checklist state.

Build order

Land terminal pane foundations first, then add Todo as the first non-terminal pane.

Deliberately avoided

No workspace tabs now

A tabbed mode is deferred until the pane-based native client exists and usage provides a reason.

Mute is not lifecycle

Hover the Codex row to reveal unmute and close where its subdued state indicator rests. The pane remains in place and retains true session state.

04 / Toolkit

Use components at the edges

gpui-component earns its place on conventional and rich-content surfaces. It accelerates native parity, but it does not become Attn's layout model or terminal engine.

Use gpui-component

  • New-session dialog and settings forms.
  • Todo controls and long-list rendering.
  • Markdown content and rich textual surfaces.
  • Diff/review controls, file lists and highlighted code text.

Keep Attn-owned

  • WorkspaceLayout and daemon-confirmed mutations.
  • Ghostty terminal rendering and PTY lifecycle.
  • Custom sidebar row actions and activity semantics.
  • Diff protocol/data model; no assumed turnkey viewer.
05 / Flow

Create, confirm, attach

Workspace membership and pane layout change only after daemon broadcasts. A pane is a daemon layout entry, never a locally manufactured terminal card or checklist.

01Choose destination

Reuse archived dialog behavior with component form controls; select directory and claude or codex.

02Register

Send register_workspace(id, title, directory) to the daemon.

03Observe

Accept workspace_registered; now the rail may render the workspace.

04Spawn agent

Send spawn_session(..., workspace_id); it creates the first agent pane.

05Display

Read WorkspaceLayout, attach its agent session PTY, create Ghostty state, render output.

06Add peer pane

Spawn into the existing workspace; render the updated layout and focus it with Cmd+Arrow.

06 / Build Order

One working slice at a time

Do not build all of the replacement app at once. Each step extends a daemon-backed client that already runs within its stated scope.

01Archive + skeleton

Keep canvas reference; initialize active crates and component root.

02Protocol v65

Connect, handshake, reconnect and read workspace layouts.

03Pane rail

Render workspace groups and focus rows from daemon state.

04Terminal panes

Port Ghostty attach, replay, resize and keyboard input.

05Create flow

Build component-backed dialog without optimistic state.

06Memory + keys

Measure workspace release policy and land navigation.

07Todo pane

Render existing daemon todo events; settle workspace aggregation.

08Shell session

Restore first-class shell creation using v65 semantics.

09Settings + Markdown

Build component-backed native rich content.

10Diff + review

Render existing daemon diff data as a native pane.

11Parity

Split controls, remaining workflows and polish.

07 / Memory

Pay only for what is visible

All workspaces

retained / metadata

IDs, titles, status and daemon layout snapshots feed the rail. Small and stable.

Selected workspace

retained / visible pane set

Ghostty buffers exist only for displayed terminal panes; Todo uses lightweight daemon data.

On switch

released / old VT

Drop the prior workspace's model set. Reattach later and reconstruct its panes from daemon replay.

Prototype decision / measure it

Default: release on switch

Lowest-memory baseline. Drop all inactive workspace Ghostty models and rebuild through daemon replay when revisited.

Compare: one warm workspace

Measure whether caching one prior pane set buys meaningful switch speed without allowing memory to scale with workspace count.

08 / Boundary

What this prototype refuses

Discarded mechanics

  • Canvas viewport, dragging, snapping or panel geometry.
  • Canvas-era shell_as_session wiring and v64 panel types.
  • Workspace surface tabs before the pane-based rewrite works.
  • Permanent workspace header bars or oversized create buttons.
  • Per-pane title bars that duplicate identity or activity state.
  • Component dock/layout ownership replacing daemon truth.
  • Optimistic local layout ownership.
  • Terminal buffers retained for inactive workspaces.

Intentional first slice

  • Sidebar is workspace-first from the initial run.
  • Creation follows register, spawn, broadcast, attach.
  • Terminal panes prove the base; Todo is the next pane kind.
  • Layout mutation controls extend WorkspaceLayout next.
  • First-class shell sessions return through v65 workspace semantics.
